# 固定翼自主起降项目：当前状态

## 1. 文档定位

本文档用于记录当前阶段项目已经确认的事实、当前默认技术口径、当前关键阻塞点，以及下一步优先事项。

本文档不替代项目总纲，也不替代阶段总结文档。其作用是：

- 让当前推进状态保持清晰
- 避免“已确认事实”和“待验证设想”混杂
- 为后续每一轮开发与实验提供统一起点

---

## 2. 当前已确认事实

截至当前，已确认如下事实：

### 2.1 PX4 + ROS + Gazebo Classic 基础仿真链路已跑通

当前已经能够在 PX4-Autopilot 工程内，以 ROS launch 方式启动：

- PX4 fixed-wing SITL
- MAVROS
- Gazebo Classic
- 自定义 `rc_runway.world`
- `plane.sdf`

这说明 fixed-wing 仿真基础链路已经建立，可作为后续视觉与控制联调平台。

### 2.2 自定义跑道 world 已接入

当前已经在 Gazebo Classic 中加载自定义跑道 world，并完成以下确认：

- `asphalt_plane` 已被移除显示，不再遮挡自定义跑道
- 自定义跑道可以在仿真场景中显示
- 跑道宽度已从 `6 m` 调整为 `4 m`

当前默认跑道文件为：

- `Tools/simulation/gazebo-classic/sitl_gazebo-classic/worlds/rc_runway.world`

### 2.3 相机图像链路已确认存在

当前 `plane.sdf` 中已经包含前视相机传感器，能够发布 ROS 图像话题。

已确认可以通过 ROS 图像可视化工具查看前视图像，这说明：

- 视觉输入链路已经具备
- 后续跑道识别与对准可以直接基于当前相机话题展开

### 2.4 当前仿真运行环境的默认口径已明确

当前已基本确认：

- PX4 / ROS / Gazebo / rqt 等系统联调，优先在 Ubuntu 系统环境中运行
- `conda` 环境不作为当前仿真主运行环境

该口径的原因是：

- ROS Noetic 与 Gazebo Classic 对系统 Python、Qt 和系统库依赖较强
- 在 `conda` 环境中直接运行 ROS GUI 和 Gazebo 更容易出现兼容冲突

### 2.5 视觉模块阶段主线已明确

当前视觉模块阶段的工程主线已经明确为：

`左右边线 -> guide_line -> 控制中间量`

即：

- 不以恢复完整三维位姿为当前阶段前提
- 先稳定提取左右边线
- 再生成跑道中轴参考引导线
- 再提取 `lookahead_point`、`heading_ref`、`e_lat_img` 等后续控制中间量

### 2.6 “QGC 一键 mission -> 视觉对准 -> PX4 起飞接管” 的系统主线已明确

当前已确认的目标链路不是“QGC 直接触发 PX4 takeoff”，而是：

`QGC Start Mission -> 视觉搜索 -> 视觉对准 -> 条件满足 -> PX4 接管起飞`

这意味着：

- 视觉模块需要承担起飞前跑道对准职责
- 系统需要增加一层对准状态机
- PX4 继续承担正式的原生起飞执行

## 3. 当前默认技术口径

### 3.1 项目总体口径

当前项目仍按以下总体结构推进：

- PX4 提供 fixed-wing 原生起飞 / 降落飞控能力
- GPS 提供基础导航参考
- 单目视觉提供跑道识别与对准几何依据
- 系统通过“识别 -> 对准 -> 起降”实现完整自主起降

### 3.2 视觉阶段口径

当前视觉阶段默认口径为：

- 经典几何 / 线检测方法作为第一版 baseline
- 左右白边线是当前最核心的结构化视觉线索
- 引导线生成比直接做中轴线检测更符合当前场景

### 3.3 交付目标口径

当前交付目标默认表述为：

> 用户在 QGroundControl 中启动 mission 后，系统先通过视觉模块完成跑道对中 / 对正；当满足几何条件后，再自动切入 PX4 原生 fixed-wing 自动起飞流程。

该目标默认针对：

- 指定机型
- 指定相机安装方式
- 指定 RC 跑道场景
- 受控环境条件

当前不将目标表述为“面向任意固定翼航模、任意跑道场景的通用产品”。

---

## 4. 当前关键阻塞点

### 4.1 视觉模块尚未形成第一版可运行 baseline

虽然当前阶段技术路线已经明确，但尚未完成：

- ROI 约束方案
- 左右边线候选提取
- 主边线筛选与拟合
- `guide_line` 生成
- `e_lat_img` / `heading_ref` 输出

因此，当前最大的实际阻塞仍是：

> 视觉模块尚未进入“可连续输出控制中间量”的状态。

### 4.2 对准控制层尚未建立

当前尚未形成明确的在线对准控制实现，包括：

- 视觉误差如何转成地面引导控制
- 对中与对正的闭环策略
- 对准完成判据的在线使用方式

也就是说，当前“视觉输出”和“PX4 起飞接管”之间仍缺少真正的中间执行层。

### 4.3 PX4 起飞接管机制仍停留在方案定义阶段

当前已经写清了推荐状态机和接管逻辑，但尚未实际实现：

- mission 启动后的视觉预对准阶段
- 视觉准入层
- 接管门槛判定
- 失败退出机制

### 4.4 真实交付边界尚需进一步固化

虽然当前已明确本项目适合“受控场景交付”，但仍需后续继续固化：

- 允许的跑道宽度与材质范围
- 相机安装高度与视场范围
- 光照与阴影容忍边界
- 机体地面转向能力边界

这些边界不固化，后续即便算法初步可跑，也难形成稳定交付定义。

## 5. 当前下一步优先事项

### 优先事项 1：先做视觉模块 baseline

下一步最优先事项应是：

- 固定当前相机话题输入
- 选定 ROI
- 完成边线候选提取
- 输出左右主边线
- 生成第一版 `guide_line`

这是后续所有对准控制与接管逻辑的前提。

### 优先事项 2：尽快定义视觉输出接口格式

在 baseline 同步推进时，应尽快固定视觉输出接口，例如：

- `runway_visible`
- `left_edge`
- `right_edge`
- `guide_line`
- `lookahead_point`
- `heading_ref`
- `e_lat_img`
- `confidence_geom`

没有固定接口，后续状态机和对准控制会反复变动。

### 优先事项 3：先做“判定型”对准，再做“控制型”对准

推荐先完成：

- 能否判定当前是否已经基本对中 / 对正

再继续推进：

- 如何利用误差做在线地面对准控制

这样可降低早期系统耦合度。

### 优先事项 4：在仿真中固化一套标准测试场景

当前建议尽快固定一套标准测试条件，包括：

- 当前 `rc_runway.world`
- 当前 `plane.sdf`
- 当前相机安装参数
- 当前默认初始位置与朝向

这样后续每次实验结果才有可比性。

`稳定视觉中间量`

做出来。
